# Pokémon Champions ダメージ計算アプリ — 本開発 引き継ぎ仕様書

**宛先:** Claude Code（本開発セッション）
**前提資料:** `requirements_v1.md`（要件定義）、`DamageCalcWireframe.jsx`（シングル試作）、`DoublesCalcWireframe.jsx`（ダブル主軸試作）、`champions-adapter.js`（検証済みエンジンアダプタ）
**最終更新:** 2026-06-04

---

## 0. このセッションのゴール

UX試作で固めた「ダブル主軸・確定数主役・3タップ」の操作モデルを、**実エンジン（@smogon/calc）の正確な計算**の上に載せた、スマホネイティブアプリの開発を開始する。本ドキュメントは検証で確定した技術前提とタスクを引き継ぐ。

---

## 1. 計算エンジンの決定（検証済み）

**採用: `@smogon/calc`（Generations.get(9) をデータ基盤に使用）。**

実パッケージを導入して以下を実測で確認済み（再現スクリプトは §6）:

1. **Champions能力システムは1行変換で厳密に載る。**
   `EV = min(252, 能力ポイント × 8)`、`IV = 31`、`Level = 50`、性格は本編式と同一。
   これでエンジン標準コンストラクタが Champions 実数値を**全域で厳密一致**で再現する。
   - 能力P=32 は EV=256→252 にクランプされるが、実数値式中の floor が差を吸収し一致する（種族値は常に偶数倍され、+95(奇)と+94(偶)の floor 結果が一致するため）。
   - 合計66能力P（=520EV相当）でもエンジンは各能力を独立計算し、本編の合計510上限でクランプしない。
2. **構築後のミューテーションは使わない。** `pokemon.stats.x` を直接書き換えると、`calculate()` が内部でクローン・再計算するため `maxHP()`（= `rawStats.hp` 駆動）に反映されず、確定数・%がズレる。必ず上記コンストラクタ経路で与えること。
3. **ダブル範囲技・テラスタル・特性・道具はエンジンが標準対応。** `Field({gameType:'Doubles'})` で範囲技に自動で×0.75。`teraType` 指定でテラスタル補正（STAB含む）が反映される。

→ 結論: **ダメージ式・乱数・確定数・KO確率・ダブル補正・テラスタルは自作しない。** Championsアダプタ（`champions-adapter.js`）でラップして使う。

### 代替案の位置づけ
`@pkmn/dmg`（@smogon/calc の後継、modによるデータ/効果上書きが非侵襲的）は、Champions差分データの上書き運用が重くなった段階で再評価する。初期は @smogon/calc + overrides で十分。

## 2. Championsアダプタ仕様

`champions-adapter.js` が中核。要点:

- `ptsToEV(p) = min(252, p*8)` … 能力ポイント→EV変換。
- `champPokemon(c)` … `{species, nature, pts, item, ability, teraType, boosts, overrides}` から `Pokemon` を生成。IV31・Lv50固定。
- `champCalc(attacker, defender, moveName, {doubles, field, moveOpts})` … 計算実行。`result.damage`(16乱数)、`result.desc()`、`result.kochance()` を返す。
- **`overrides`** で species 単位の Champions 差分（baseStats・types・特性など）を注入できるフックを用意済み。

注意: エンジンのデータは**英語名**（'Garchomp', 'Earthquake'）。アプリは日本語UIなので、**JP↔EN名の対応表をデータ層に持つ**必要がある（§3）。

---

## 3. データ層の設計（本プロジェクト最大の継続コスト）

二層構成にする。

- **ベース層:** @smogon/calc 内蔵データ（種族値・タイプ・特性・技・道具）＋ JP↔EN名マッピング。種族・技・特性・道具・テラスの日本語表示はここで解決。
- **Champions差分層（手動整備・継続パッチ必須）:**
  - 技威力の Champions 調整値
  - 内定ロスター（出られるポケモン）・メガシンカ可否
  - ゼンブイリングなど Champions 固有要素の挙動
  - 特性/道具の挙動差分
  差分は `overrides` および技データの上書きとして適用。**ゲーム更新に追随する更新フローを最初から設計**（要件 N-6: アプリ更新を待たずに差分配信できる構成）。
- **使用率データ（任意・対策分析用）:** 外部の使用率・素早さ帯データを取得できれば、デフォルト配分や脅威提示に活用。取得手段は要調査。

データ精度は実機照合で担保する（競合も実機検証で精度を出している）。

---

## 4. 技術スタックの推奨

- **React Native + TypeScript** を第一候補とする。計算エンジン（@smogon/calc）と試作UIがJS/TS資産であり、ロジックをそのまま共有できるため、ブリッジを挟むFlutter等より素直。
- 試作（`DoublesCalcWireframe.jsx`）のコンポーネント構造・状態モデル・配色/タイポグラフィをRNへ移植する。インタラクション仕様は試作がそのままUX仕様書。
- 状態管理は軽量なもの（Zustand等）で可。オフライン動作（要件 N-3）のためデータはローカル同梱＋差分更新。

---

## 5. 推奨マイルストーン

1. **プロジェクト雛形**（RN+TS）、`champions-adapter.js` をTS化して組み込み、ユニットテスト（既知の Champions 実数値・サンプル確定数を assert）を整備。
2. **データ層の最小実装**: ベース層＋JP↔EN名マップ＋差分層の枠組み（最初は主要ポケモン/技のみ）。
3. **ダブル計算画面**: 試作の2v2盤面・複数対象同時表示・味方巻き込み・合算KOを、アダプタの実計算に接続。
4. **耐久/素早さ調整・タイプ相性/対策**（要件 F-9〜F-14）。
5. **プリセット保存（F-8）・全技ダメ計（F-4）・両方向計算（F-5）**。
6. **差別化の決め手**: ゲーム内パーティのインポート（チームコード等）を調査・試作。手入力削減は競合との最大の差。

## 6. 検証の再現（参考）

`champions-adapter.js` 自体が自己テスト付き（`node champions-adapter.js`）。確認した代表値:

- ガブリアス(A特化32/いじっぱり) じしん vs カバルドン(HB特化) … 乱数 61–73、対215HPで約28–34%、確定3発相当。
- 同条件を `gameType:'Doubles'`（範囲）にすると 46–55 に低下（×0.75 をエンジンが適用）。
- テラスタル『じめん』指定でじしんが Tera STAB により増加。
- 能力配分 32/32/2（合計66P=520EV相当）でも hp/def/spd 各値が Champions 式と厳密一致（合計510でクランプされない）。

## 7. 引き継ぐ未解決事項

- ゼンブイリング・テラスタル・メガの計算上の正確な扱いは**実機検証**で最終確定する（差分層に反映）。
- 使用率データの取得可否・手段。
- マネタイズ方針（広告なしを差別化に使うか）と、収益化アプリとしての権利面の確認（要件 §9 リスク表参照）。
- 計算式の細部（連続技の独立乱数、補正適用順）はエンジンが正しく扱うため自作不要だが、Champions差分が本編と挙動を変える箇所がないか差分層整備時に確認。
